## Issues Log

### BLOCKERS (2)

#### 1. Test Coverage: 0% (Required: 80%)

**Severity**: BLOCKER  
**Detected By**: Deployment readiness tool  
**Context**: Tool expects application-style unit/integration tests  

**Root Cause**:
- Infrastructure automation projects (Ansible playbooks) typically don't have traditional unit tests
- Deployment readiness tool assumes application testing patterns (Jest, Playwright, etc.)
- No test framework configured (molecule, ansible-test)

**Impact**:
- Blocks automated deployment approval
- Score reduced to 65/100

**Recommended Fix**:
- **Option 1**: Implement Ansible testing framework (molecule for role testing)
- **Option 2**: Add infrastructure project exemption to deployment_readiness tool
- **Option 3**: Create separate validation tier for infrastructure vs application projects

**Next Action**: Determine testing strategy for infrastructure automation (create ADR-0036)

## Key Lessons Learned

### 1. What Went Better Than Expected

✅ **Complete ADR-0035 Implementation**:
- All 8 phases of Agent-Based Installer integration complete
- Role-based architecture (ADR-0024) properly applied throughout
- Clean integration with existing infrastructure (common_vm role)

✅ **Infrastructure Stability**:
- 8/10 infrastructure checks passed without intervention
- Hypervisor, networking, storage, and DNS all operational
- 6 VMs running (VyOS, registry, AAP 3-node cluster)

✅ **Documentation Quality**:
- 4 comprehensive documentation files delivered
- Clear KVM vs bare metal workflow distinction
- Tutorial, how-to, and reference docs all present

✅ **Multi-Registry Abstraction**:
- Quay, Harbor, JFrog support implemented cleanly
- Registry-agnostic templating working as designed

### 2. What Was Harder Than Expected

⚠️ **Testing Mismatch**:
- Deployment readiness tool expectations don't align with infrastructure automation patterns
- Ansible projects don't typically have unit/integration tests like application code
- No clear guidance on testing strategy for infrastructure automation

⚠️ **User Prerequisite Validation**:
- Pull secret requirement documented but not validated before deployment
- No preflight check for user-provided artifacts
- Late detection of missing prerequisites wastes validation time

⚠️ **Health Check Reliability**:
- Quay health endpoint returned unexpected null despite service running
- Difficulty distinguishing between "starting up" and "unhealthy"
- Need functional tests (image push/pull) not just health checks

### 3. Issues Not Anticipated by Any ADR

**Infrastructure Testing Strategy Gap**:
- ADRs document features and architecture but not testing requirements
- No ADR addresses whether infrastructure automation needs unit tests
- Deployment readiness tool assumes application testing patterns

**User Prerequisites Validation Gap**:
- ADR-0035 assumes deployment prerequisites exist but doesn't validate them
- No framework for checking user-provided artifacts (pull secrets, credentials, keys)
- Late failure wastes deployment time

**Documentation Count Inconsistency**:
- Phase 8 plan mentioned 12 cluster configs, implementation created 8
- Not tracked through ADR updates during implementation
- Minor issue but highlights need for implementation ↔ documentation sync

### 4. Workarounds Applied

**Test Coverage Blocker**:
- **Workaround**: Continued validation to identify all issues despite blocker
- **Should Become Permanent**: No - either implement testing framework or exempt infrastructure projects from test requirement
- **Action**: Create ADR-0036 to document testing strategy decision

**Pull Secret Missing**:
- **Workaround**: None - hard blocker
- **Should Become Permanent**: Add preflight validation playbook
- **Action**: Create ADR-0037 for prerequisite validation framework

### 5. ADR Assumptions Challenged

**ADR-0035: Agent-Based Installer Adoption**:
- ✅ Correct: Agent-Based Installer is the right approach
- ✅ Correct: Multi-registry support required
- ❌ Incomplete: Doesn't document user prerequisite validation
- ❌ Incomplete: Doesn't address testing strategy for infrastructure code

**Documentation Accuracy**:
- Plan vs implementation mismatch (12 configs planned, 8 delivered)
- Need tighter sync between ADR planning phases and actual implementation

---

**ADR-0037: Deployment Prerequisites Validation Framework**

**Decision**: How should user-provided prerequisites be validated before automation execution?

**Context**:
- Pull secrets, AWS credentials, SSH keys are user-provided
- Missing prerequisites cause late failures (wasted time)
- No unified preflight validation framework exists

**Recommendation**: Create preflight validation playbook that:
- Checks file existence (/root/pull-secret.json, ~/.aws/credentials, ~/.ssh/id_rsa.pub)
- Validates file format (JSON valid, credentials parseable)
- Tests connectivity (registry reachable, AWS API accessible)
- Runs before main automation playbooks
- Fails fast with clear error messages

## Recommended ADR Updates

### ADR-0035: Adopt OpenShift Agent-Based Installer

**Section to Add**: "Prerequisites Validation"

```markdown
## Prerequisites Validation

Before running the deployment playbook, users must:

1. **Obtain Red Hat Pull Secret**:
   - Download from https://console.redhat.com/openshift/install/pull-secret
   - Save to /root/pull-secret.json
   - Validate: `cat /root/pull-secret.json | jq .auths`

2. **Verify Hypervisor Resources**:
   - Run preflight check: `ansible-playbook playbooks/preflight-openshift-cluster.yml`
   - Check output for resource adequacy

3. **Configure DNS Provider** (if using Route53):
   - AWS credentials in ~/.aws/credentials
   - Test: `aws route53 list-hosted-zones`

**Preflight Playbook**:
- `playbooks/preflight-openshift-cluster.yml` validates all prerequisites
- Run before `deploy-openshift-cluster.yml`
- Fails fast with actionable error messages
```

## Conclusion

**v1.3.0 Implementation Quality**: Excellent (10/10 application checks passed)  
**Deployment Readiness**: Blocked (2 critical prerequisites not met)  
**Root Cause**: User prerequisites validation gap, testing strategy mismatch

**Primary Learning**: Infrastructure automation projects require different validation patterns than application code. Need to:
1. Validate user prerequisites before deployment (not during)
2. Define testing strategy appropriate for infrastructure code
3. Update tooling to support infrastructure project patterns

**Release Recommendation**: Do not proceed to production until:
- Pull secret provided by user
- Test coverage requirement decision documented in ADR-0036
- Preflight validation framework implemented (ADR-0037)

**Overall Assessment**: Implementation is complete and high-quality. Blockers are environmental/process issues, not code defects. With prerequisite validation framework, v1.3.0 will be production-ready.
